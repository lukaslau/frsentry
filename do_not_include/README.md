# frsentry — Developer Documentation

PrestaShop Sentry integration module. Monitors PHP errors, exceptions, DB failures,
and frontend JS issues by forwarding them to a Sentry instance.

## Why PHP-Scoper?

`ps_mbo` (PrestaShop marketplace module) ships its own copy of `sentry/sentry` and
registers its autoloader with `prepend = true`. Without isolation, calling
`\Sentry\captureException()` resolves to ps_mbo's functions which use
`HttplugClient` → `DecoderPlugin` → `FilteredStream`. On PHP 8.4 this causes a
fatal error (`seek(int, int)` incompatible with untyped `StreamInterface::seek`).

**Solution:** PHP-Scoper rewrites our copy of the Sentry SDK so all its classes and
functions live under the `FrSentry\Sentry\*` namespace. Our code calls
`\FrSentry\Sentry\captureException()` which always resolves to our isolated copy,
regardless of what other modules load.

Only `vendor/sentry/sentry/src/` is scoped. External dependencies
(Symfony, Psr\Log, etc.) are intentionally left unscoped — they don't conflict and
are shared from `vendor/`.

---

## Autoloader boot order in frsentry.php

```php
// 1. Load scoped Sentry SDK (FrSentry\Sentry\* classes)
require_once __DIR__ . '/libs/sentry/autoload.php';

// 2. Load deps: Symfony\OptionsResolver, Psr\Log, etc.
require_once __DIR__ . '/vendor/autoload.php';
```

`sentry/sentry` is `require-dev` only, so production `vendor/` never contains the
unscoped SDK — no risk of `\Sentry\*` function redeclaration conflicts with ps_mbo.

## composer.json — dependency pinning

`sentry/sentry` supports PHP `^7.2|^8.0`, but its transitive dependencies resolve
to PHP 8.1+ versions if left unpinned. These pins keep everything on PHP 7.2+:

| Package | Pinned to | Why |
|---|---|---|
| `symfony/options-resolver` | `^5.4` | v7.x requires PHP ≥8.2 |
| `symfony/deprecation-contracts` | `^2.5` | v3.x requires PHP ≥8.1 |
| `psr/log` | `^1.0` | v3.x requires PHP ≥8.0 |
| `jean85/pretty-package-versions` | `^1.6` | v2.x requires PHP ≥7.4 |

`sentry/sentry` itself is in `require-dev` only — it is needed to generate
`libs/sentry/` via PHP-Scoper, but is not required at runtime.

After any `composer update`, verify no package requires PHP 8+ exclusively:
```bash
python -c "
import json
with open('composer.lock') as f:
    lock = json.load(f)
for p in lock.get('packages', []):
    php = p.get('require', {}).get('php', '')
    if '8.1' in php or '8.2' in php:
        print(p['name'], p['version'], '->', php)
"
```

## Scripts

### `do_not_include/dev.bat`
Installs all dependencies including dev tools.
```
do_not_include\dev.bat
```

### `do_not_include/cs-fix.bat`
Runs PHP CS Fixer on the project source.
```
do_not_include\cs-fix.bat
```

### `do_not_include/rebuild-sentry.bat`
Rebuilds `libs/sentry/` from scratch. Run this after updating `sentry/sentry` version.
```
do_not_include\rebuild-sentry.bat
```
Steps performed:
1. PHP-Scoper rewrites `vendor/sentry/sentry/src/` → `libs/sentry/` under `FrSentry\` prefix
2. `composer dump-autoload` regenerates `libs/sentry/autoload.php`

### `do_not_include/release.bat`
Builds a production release into `release/frsentry/`.
```
do_not_include\release.bat         # copies files to release/frsentry/
do_not_include\release.bat /zip    # also creates frsentry.zip
```
Steps performed:
1. PHP CS Fixer
2. `composer install --no-dev --optimize-autoloader`
3. Copies module files to `release/frsentry/` (excludes dev files)
4. Optionally creates `frsentry.zip` (with `/zip` flag)
5. Restores dev dependencies

Excluded from release: `.git`, `.claude`, `do_not_include`, `*.bat`, `*.zip`,
`.gitignore`, `composer.lock`, `.php-cs-fixer.cache`.

## Updating the Sentry SDK

1. Update version in `composer.json` (`require-dev`)
2. Run `composer update sentry/sentry`
3. Run `do_not_include\rebuild-sentry.bat`
4. Verify the autoload hash — check `vendor/composer/autoload_files.php` for the
   `sentry/sentry/src/functions.php` entry and update the hash in `frsentry.php`
   if it changed
